## 2. 开发方法论与核心原则

### 2.1 学习方式

1. **先手写原生理解本质 → 再学框架设计思想 → 最后回归自研改进**
   - 例：先手写 ReAct 循环（`agent_core/`），再 LangGraph 重构（`langgraph_agent/`），对比学习声明式 vs 命令式
2. **从 Claude Code 源码学习工程实践**，但只做参考，不被其绑架
3. **极简依赖**：只装必须的 SDK（如 `zhipuai`），不装框架（如 LangChain）

### 2.2 工程决策原则

- **没有 baseline 的实验不做**
- **没有离线评估的模型不上线**
- **推理服务必须有降级策略**
- **不在生产环境用未验证的模型**
- **务实优先，选最适合业务场景的方案，不追最新论文**

### 2.3 数据驱动

- 所有 Token 估算基于实测，不使用假设数字
- 中文 1.4 tokens/字，英文 0.25 tokens/字符
- `SimpleTokenCounter` 误差 ±20%，作为预算管理的近似值
- API 返回的 `input_tokens` 是唯一权威来源

### 2.4 对话代理（AI 助手）协作规范

- **只看** `/Users/fanyunxu/Desktop/myproject/ailearning/claude-code-analysis/` 下的 Claude Code 源码（**非 minified**）
- **禁止看** `/usr/local/lib/node_modules/@anthropic-ai/claude-code/` 下的 minified `cli.js`
- 代码改动前先读现有代码，理解上下文再动
- 不确定时明确说"不确定"，**绝对不凭印象编造**

## 6. UI 开发规范（Streamlit）

### 6.1 核心陷阱

1. **`session_state` 是惰性初始化**
   - 配置变更（provider/model/system_prompt/max_turns）需要显式检测并触发 Agent 重建
   - 使用 `last_agent_config` 缓存对比四个关键字段（类似 React `useEffect` dependency array）

2. **同名变量承载不同类型是事故根源**
   - P5 重构中 `turn_thinking` 同时承载 `dict[int, str]` 和 `int` → 两次连续 crash
   - 解决方案：改名消歧（`turn_thinking` → `turn_thinking_tokens`）+ 各自独立初始化

3. **变量作用域与初始化**
   - 在条件分支外声明变量（如 `thinking_text = ""`），避免分支内定义的变量在分支外引用时 `NameError`

4. **F5 刷新时的状态恢复**
   - Agent 重建时需从 jsonl 恢复 `chunk.usage` baseline
   - 否则从增量路径切换到全量路径，Token 数字跳变（偏高 10-50%）

### 6.2 UI 数字语义

- **左侧 sidebar**：当前对话累计 Token（baseline + 增量估算）
- **压缩判断**：与左侧共用同一个 `_estimate_used_tokens()` 返回值
- **API 返回的 `input_tokens`**：仅 LLM 实际收到的消息数量（不含 assistant 自身）

## 7. LLM 集成规范

### 7.1 多 Provider 支持

| Provider | System Prompt 传递方式 | Thinking 支持 | Tool Choice 支持 |
|----------|------------------------|---------------|------------------|
| Anthropic | 顶层 `system` 参数 | ✅ (chunk.thinking_delta) | ✅ |
| OpenAI/Zhipu | messages 首个 `system` role | ❌ | 待验证 |
| GLM-4.6/5.1 | messages 首个 `system` role | ❌（可用 Prompt hack） | 待验证 |

### 7.2 关键约束

1. **Anthropic system prompt 必须作为顶层参数**，不能混入 messages
2. **GLM 不支持 Claude 风格 thinking blocks API**，但可通过 Prompt 让 GLM 输出【思考】标记
3. **GLM cache 机制**：自动 prefix matching，**不需要** `cache_control` marker（实测确认）
4. **GLM-5.1 流式偶发断流**："request ended without sending any chunks"
5. **MiniMax-M3 偶发 400**：tool result 的 tool id 找不到

### 7.3 Cache 命中规则（GLM）

- **Fork 压缩**：传递 `parent_system` + `parent_tools` + `parent_messages` → 前缀一致 → cache 命中 100%（实测 25,664/25,684，§4.5.8）
- **旧模式压缩**：短 system prompt + 无 tools → cache 0%
- **skipCacheWrite**：Anthropic API 参数，GLM 无等价物，但 GLM 自动 prefix matching 天然省 cache

## 8. 上下文压缩系统规则

### 8.1 压缩触发条件

- Token 用量 > 窗口的 80% 时触发
- `compress_all_histories()` 阈值：30min + 1天（待调整为更长时间）
- `max_turns`：硬上限，超出的对话 turn 丢弃

### 8.2 Preserved Head 算法（v4.1，2026-06-18 修复）

```
从最近到最早遍历 turns：
  if 已选够 max_turns → break（硬停）
  if total + turn_tokens > max_total_tokens → break（硬 stop, P1-2 修复后无任何例外）
  selected_turns.insert(0, turn)
  total += turn_tokens
```

- **v4.1 vs v4 关键差异**（P1-2 修复）：v4 有"第一个 turn 必含"反模式，让单 turn 灌水（>4000 tok）绕过硬限制。v4.1 取消这个例外，零截断零例外。
- **退化场景**：第一个 turn 超 budget → preserved head 为空，由 summary 兜底
- **参数**：`max_total_tokens=4000`，`max_turns=3`

### 8.3 压缩提示词设计

- **单一 `COMPACT_INSTRUCTION` 模板**（维护一份就好）
- **两个 builder 函数**：
  - `build_compact_user_prompt(text)`：旧模式，对话内容嵌在 user prompt 末尾
  - `build_compact_fork_prompt()`：Fork 模式，纯指令，对话由 `parent_messages` 传递
- **输出格式**：`<analysis>` + `<summary>`（4 段结构：用户目标/关键决策/当前状态/待办事项）

### 8.4 PTL 防御机制

- 压缩请求超限时截断 20% 最旧消息，最多重试 3 次
- 每次都剥掉最外层旧消息（"剥洋葱"）

### 8.5 Token 估算架构

| 层级 | 精度 | 速度 | 使用场景 |
|------|------|------|----------|
| API `input_tokens` | ±0% | O(1) | 基准值 |
| 增量估算（SimpleTokenCounter） | ±20% | O(ΔN) | 新增消息 |
| 全量估算（SimpleTokenCounter） | ±20% | O(N) | 基准失效时兜底 |

**增量估算流程**：
1. LLM 流式响应末尾捕获 `chunk.usage.input_tokens` → `set_baseline(tokens, msg_count)`
2. 后续每轮：`used_tokens = baseline + count(new_messages)`
3. 基准失效时（首次启动、压缩后、F5 刷新恢复失败）：退化为全量估算

**Baseline 持久化**：
- `chunk.usage` 序列化为 `asdict()` 存入 jsonl entry
- Agent 重建时：`read_tail(64KB)` O(1) 恢复 → miss 时 `read_entries()` O(n) fallback
- `baseline_msg_count = len(messages) - 1`（因为最后一条 assistant 消息的 input_tokens 不含它自己）

- ❌ **不编造**：不确定的必须说"不确定"，绝不能凭印象编一个听起来合理的说法
- ❌ **不读 minified 源码**（`cli.js`），只看分析版
- ❌ **不跳过测试**：每次改动后必须跑 pytest
- ❌ **不同名变量承载不同类型**：出现就改名消歧

### 11.2 设计铁律

- ✅ Fork 模式和旧模式的压缩提示词**维护一份**
- ✅ Preserved Head **硬 stop**，不软超 budget
- ✅ Fork 压缩**必须传 tools** 保 cache 命中
- ✅ Baseline 持久化用 `asdict(usage)`，不加额外字段
- ✅ **回退路径与主路径同等严谨**：不能因为"反正不走"就放水。`SimpleTokenCounter` v2 的 5 类启发式回退（中文 0.45/英文 0.22/代码 0.33/数字 0.45）经与 tiktoken o200k_base 对照校准，偏差控制在 ±15%，与主路径精度持平
