### 08 Prompt Cache 实战：让 Agent 的 LLM 账单砍掉 90%

**核心主题**：LLM Prompt Cache 机制与工程优化

**内容要点**：
- 什么是 Prompt Cache：服务端 KV-Cache 复用，命中读免费（省 90%），首次写贵 25%
- 四大模型的 cache 行为差异：
  - Anthropic：显式 `cache_control: {type: "ephemeral"}` 标记，精确控制缓存边界
  - GLM：自动 prefix matching，不需要 `cache_control`，但行为不稳定
  - OpenAI：implicit caching，应用层无需干预
  - MiniMax：implicit caching，实测 61.3% 命中率
- `cache_namespace` 设计：同 namespace 跨调用共享 cache，不同 namespace 隔离
- GLM A/B 测试实战：curl 验证 + `glm_cache_abtest.py` 脚本，实测 94.7% 命中率
- Fork 压缩的 cache 复用：`skipCacheWrite=true`，字节级一致命中 100%
- 13 字段 Cache Key 拼接机制（Claude Code 源码学习）：system / tools / model / agentId 等
- `cache_safe_params` 设计：记忆提取 Agent 的 prompt 结构稳定，多次调用共享前缀
- Cache miss 诊断：`promptCacheBreakDetection` 机制，13 字段任一变化 → cache 必然 miss
- 实战踩坑：空 system_prompt 注入导致 cache prefix 对齐被破坏
- 成本量化：cache 命中率从 0% 提升到 80%，日成本从 $27 降到 $13（省 51%）

**预期篇幅**：5500 字

## 第四篇：上下文管理篇（3 篇 · 深入细节）

> 目标：解决"长对话怎么不断"这个 Agent 的生死问题

### 13 长对话压缩：compact 摘要的工程实现

**核心主题**：自动压缩机制与防御设计

**内容要点**：
- 自动触发机制：`shouldAutoCompact()` 每轮检查 token 阈值
- PTL 防御：压缩请求本身也可能超限，"剥洋葱"重试截断 20% 最旧消息
- 熔断器设计：连续失败 3 次停止重试，避免死循环
- Preserved Head：保留最近 6 条原始消息保证上下文连续性
- 防漂移：摘要必须逐字引用用户消息，`<analysis>` + `<summary>` 双标签
- 软标记不删除：压缩后旧消息不物理删除，用 boundary 标记，可审计可回退
- compact window 三约束：cap(40K) + min-tokens(10K) + min-text-blocks(5)
- API 不变量保护：tool_use/tool_result 配对不被切断

**预期篇幅**：5000 字

### 14 JSONL 会话持久化：为什么不用数据库存对话

**核心主题**：会话存储方案选型与实现

**内容要点**：
- 为什么选 JSONL 而不是 SQLite/PostgreSQL：Append-only 天然适合
- JSONL 的 O(1) 读取方案：不需要解析整个文件
- 会话创建 / 切换 / 恢复的完整流程
- SQLite 的角色：只存 meta（cursor、标题、时间），不存对话内容
- 灾难恢复：JSONL 文件损坏的 `.recovery-backup` 机制
- 并发安全：file lock 防止多进程写入冲突
- 性能对比：JSONL vs SQLite 在不同场景下的读写性能

**预期篇幅**：4000 字

### 15 Token 估算陷阱：tiktoken 在 GLM 上为什么不准

**核心主题**：Token 估算的工程实践

**内容要点**：
- tiktoken 的工作原理与局限：只对 OpenAI 模型准确
- GLM / Claude 的 tokenizer 差异：同一个中文句子，不同模型 token 数差 2 倍
- 实测数据：中文 1.4 tokens/字，英文 0.25 tokens/字符的来源
- `SimpleTokenCounter` 的 ±20% 误差：预算管理的近似值
- API 返回的 `input_tokens` 才是唯一权威来源
- 估算与实际偏差对 autoCompact 触发时机的影响
- 工程策略：估算用于预防，实际值用于校准
